# 第1章 系统需求分析

## 1.1 需求描述

### 1.1.1 普通用户需求

随着移动互联网与在线餐饮服务的快速发展，普通用户对于网上订餐系统的核心需求可归纳为以下几个方面：

**（1）便捷的浏览与搜索需求。** 用户希望能够快速浏览平台上的商家和菜品信息，通过分类导航（如快餐简餐、奶茶饮品、川湘辣味、粤菜海鲜、甜品小吃、早餐早点、面食粥品、轻食沙拉等）快速定位目标商家，并通过关键词搜索功能精确查找所需菜品。首页应提供轮播活动Banner、热门商家推荐、限时特价菜品展示等信息，帮助用户发现优质商家与优惠活动。

**（2）流畅的进店点餐体验。** 用户期望能够进入商家店铺后，通过左侧菜品分类栏与右侧菜品列表快速浏览本店全部菜品，支持菜品规格选择、数量增减、一键加入购物车等操作。购物车应实时显示已选商品、合计金额，并支持底部悬浮购物车弹窗，方便用户随时查看和调整已选商品。

**（3）安全可靠的下单结算流程。** 用户在结算时需要填写收货人姓名、手机号码、收货地址等信息，并选择支付方式（微信支付或支付宝支付）。系统应支持微信支付和支付宝支付两种主流支付方式，订单提交后生成唯一订单编号，并自动扣减菜品库存。同时，系统应支持跨店铺结算拦截，当购物车中存在不同店铺商品时，提示用户不可跨店结算。

**（4）完整的订单管理与追踪需求。** 用户期望能够查看全部历史订单列表，支持按默认排序或按时间排序。在订单详情页中，用户可以查看订单状态（待支付、待接单、已接单、制作中、配送中、已完成、已取消）、商品明细、配送信息等。用户还可以在规定时限内执行确认收货、取消订单等操作，并查看配送员的实时位置信息。

**（5）便捷的沟通与客服需求。** 用户希望能够与商家或管理员进行在线消息沟通，咨询订单相关问题。同时，系统提供AI智能客服功能，用户可通过悬浮聊天窗口随时提问，系统自动读取用户订单和配送上下文信息，提供关于订单查询、取消规则、配送进度等问题的智能解答。

### 1.1.2 商家需求

商家作为平台的核心服务提供者，其核心需求主要包括以下方面：

**（1）店铺信息管理需求。** 商家需要能够设置和修改店铺基本信息，包括店铺名称、联系电话、店铺分类（品类归属）等。店铺分类信息将影响用户在首页通过分类导航查找商家时的展示结果，因此商家需要准确选择自身所属品类。

**（2）菜品全生命周期管理需求。** 商家需要对本店菜品进行完整的管理操作，包括新增菜品（填写名称、价格、库存、分类、描述、上传图片）、编辑菜品信息、上下架菜品、设置特价菜品、删除菜品等。菜品管理页面应支持按名称搜索、按分类筛选、分页展示等功能，方便商家高效管理大量菜品。

**（3）订单实时处理需求。** 商家需要实时查看本店收到的订单，并根据订单状态执行相应操作：接单（将"待接单"状态更新为"已接单"）、拒单（将"待接单"状态更新为"已取消"并填写原因）、开始制作（将"已接单"更新为"制作中"）、出餐（将"制作中"更新为"配送中"）、取消订单（在已接单或制作中状态下可取消，需填写原因）。商家还可在配送过程中实时更新配送位置信息。

**（4）沟通协作需求。** 商家需要与平台管理员保持沟通渠道，接收管理员的通知和指令。同时，商家也需要与用户进行消息交互，回复用户关于订单、菜品等问题的咨询。

### 1.1.3 管理员需求

管理员作为平台的运营管理者，承担着平台全局管控的职责，其核心需求包括：

**（1）平台数据监控需求。** 管理员需要在后台首页查看平台整体运营数据，包括入驻商家数量、注册用户数量、全平台菜品数量、全平台订单数量等核心指标，并支持一键跳转至各管理模块。

**（2）商家审核与管理需求。** 管理员需要对新注册的商家进行入驻审核（商家注册时状态为"审核中"，审核通过后状态变更为"正常"），对已入驻商家进行分类管理（设置店铺所属品类）、封禁/启用操作。管理员可以查看商家详情，包括该商家的菜品列表和订单列表。

**（3）用户管理需求。** 管理员需要对平台所有用户进行管理，包括查看用户列表、搜索用户、禁用/启用用户账号、重置用户密码、变更用户角色等操作。对于待审核的商家入驻申请，管理员需在用户管理页面进行专项处理。

**（4）菜品与分类管理需求。** 管理员拥有对全平台菜品的管理权限，可以查看所有商家的菜品、进行搜索和分页浏览。管理员还可以管理菜品分类体系，包括新增分类、编辑分类名称、删除分类（需校验分类下是否关联菜品）等。

**（5）订单全局管控需求。** 管理员可以查看全平台所有订单，支持按商家筛选、按用户名搜索、按默认排序或时间排序。管理员拥有订单状态的最高修改权限，可以强制修改任意订单的状态（包括强制取消），但执行操作时必须填写操作原因，所有状态变更均记录操作日志。

**（6）平台内容运营需求。** 管理员需要管理首页轮播Banner图片，包括上传图片、设置标题和副标题、配置点击跳转链接、调整排序、启用/禁用等操作，以支持平台活动推广和运营需求。

## 1.2 系统核心业务分析

### 1.2.1 整体业务流程

本系统围绕"用户在线点餐—商家接单制作—配送送达"的核心业务链路展开，整体业务流程如下：

**（1）用户注册与登录流程。** 用户访问系统首页后，可通过注册页面创建账号。注册时需填写用户名、密码、手机号等信息，系统对密码进行BCrypt加密后存储。商家注册时还需填写店铺名称和店铺分类，注册后状态为"审核中"，需等待管理员审核通过后方可登录。用户登录时，系统根据角色（普通用户/商家/管理员）跳转至对应的首页。

**（2）菜品浏览与进店点餐流程。** 用户在首页浏览热门商家和特价菜品，点击商家卡片进入店铺主页。店铺页面采用左侧分类栏+右侧菜品列表的美团同款布局，用户可通过左侧分类栏快速定位菜品，点击"+"按钮将菜品加入购物车。购物车绑定当前店铺，底部悬浮购物车弹窗实时显示已选商品和合计金额。

**（3）购物车与结算流程。** 用户在店铺页面完成选餐后，点击底部购物车弹窗中的"去结算"按钮，进入结算页面。系统校验购物车商品是否属于同一店铺，若存在跨店商品则拦截并提示。用户填写收货信息并选择支付方式后提交订单，系统在事务中完成库存校验、订单创建、库存扣减、购物车清空等操作。

**（4）支付与订单状态流转流程。** 订单提交后进入"待支付"状态，用户选择支付方式完成支付后，订单状态变更为"待商家接单"。商家接单后状态变更为"已接单"，开始制作后变更为"制作中"，出餐后变更为"配送中"。用户确认收货后订单状态变更为"已完成"。整个过程中，系统记录每次状态变更的操作日志。

**（5）配送追踪流程。** 订单进入"配送中"状态后，商家或管理员可以实时更新配送员的位置信息和配送状态（待配送/配送中/已送达）。用户在订单详情页可通过弹窗查看配送员的当前位置描述。配送员标记"已送达"后，配送信息锁定不可修改。

### 1.2.2 核心功能模块划分

根据业务分析，本系统划分为以下核心功能模块：

| 模块编号 | 模块名称 | 主要功能 |
|----------|----------|----------|
| M1 | 用户管理模块 | 注册、登录、个人中心、密码修改、角色权限控制 |
| M2 | 商家管理模块 | 商家入驻、审核、店铺设置、分类管理、封禁启用 |
| M3 | 菜品管理模块 | 菜品CRUD、上下架、图片上传、分类管理、特价设置 |
| M4 | 购物车模块 | 加购、数量修改、删除、按店铺分组、跨店拦截 |
| M5 | 订单管理模块 | 下单、支付、状态流转、搜索排序、跨店校验 |
| M6 | 配送管理模块 | 配送状态更新、位置追踪、信息查询 |
| M7 | 消息通信模块 | 用户-商家消息、商家-管理员消息、管理员-用户消息 |
| M8 | 首页展示模块 | 轮播Banner、分类导航、热门商家、特价菜品 |
| M9 | AI客服模块 | 智能问答、订单上下文读取、配送进度查询 |
| M10 | 系统管理模块 | 数据统计、轮播管理、用户管理、权限控制 |

### 1.2.3 业务活动图描述

以下以文字形式描述各核心业务的活动图，供手绘参考。

**活动图1：用户点餐下单流程**

```
[开始] → 用户登录系统 → 浏览首页/搜索商家 → 进入商家店铺
  → 浏览菜品列表 → 选择菜品加入购物车 → 继续选餐/去结算
  → 进入结算页 → 填写收货信息 → 选择支付方式 → 提交订单
  → 系统校验库存 → [库存充足？]
    → 是：创建订单 → 扣减库存 → 清空购物车 → 跳转支付页 → 完成支付 → [结束]
    → 否：提示库存不足 → 返回购物车 → [结束]
```

**活动图2：商家接单处理流程**

```
[开始] → 商家登录后台 → 查看订单列表 → 选择待接单订单
  → [接单/拒单？]
    → 接单：状态变更为"已接单" → 记录接单时间 → 等待制作
    → 拒单：填写拒单原因 → 状态变更为"已取消" → 记录日志 → [结束]
  → 开始制作 → 状态变更为"制作中" → 制作完成出餐
  → 状态变更为"配送中" → 更新配送位置 → 等待用户确认收货
  → [结束]
```

**活动图3：管理员审核商家流程**

```
[开始] → 管理员登录后台 → 查看待审核商家列表 → 选择待审核商家
  → 查看商家信息 → [审核通过/拒绝？]
    → 通过：状态变更为"正常" → 通知商家 → [结束]
    → 拒绝：状态变更为"禁用" → 通知商家 → [结束]
```

#### 用例1：用户注册

| 项目 | 内容 |
|------|------|
| 用例名称 | 用户注册 |
| 用例编号 | UC-001 |
| 参与者 | 普通用户 |
| 前置条件 | 用户未登录，已访问注册页面 |
| 后置条件 | 注册成功后跳转至登录页；商家注册后状态为"审核中" |
| 基本流 | 1. 用户填写用户名、密码、手机号等信息；2. 点击"注册"按钮；3. 系统校验用户名唯一性、密码强度、手机号格式；4. 系统对密码进行BCrypt加密后存入数据库；5. 普通用户注册成功，跳转登录页 |
| 异常流 | 1a. 用户名已存在：提示"用户名已存在"；1b. 手机号为空：提示"手机号不能为空"；1c. 密码为空：提示"密码不能为空" |
