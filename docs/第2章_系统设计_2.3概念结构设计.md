# 第2章 系统设计（续）

## 2.3 数据库设计

### 2.3.1 概念结构设计

#### 一、实体及其属性

本系统共识别出 **10个实体**，各实体及其属性如下（标注 **PK** 为标识符/主键，标注 **FK** 为外键）：

---

**实体1：用户（User）**

| 属性名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| **user_id** | INT | **PK**，自增 | 用户唯一标识 |
| username | VARCHAR(30) | UNIQUE, NOT NULL | 登录用户名 |
| password | VARCHAR(255) | NOT NULL | BCrypt加密密码 |
| real_name | VARCHAR(20) | | 真实姓名（商家即店铺名称） |
| phone | VARCHAR(15) | | 手机号码 |
| email | VARCHAR(50) | | 电子邮箱 |
| address | VARCHAR(200) | | 收货地址 |
| role | TINYINT | NOT NULL | 角色类型：0=普通用户, 1=管理员, 2=商家 |
| status | TINYINT | NOT NULL | 账号状态：0=正常, 1=禁用, 2=审核中 |
| shop_category | VARCHAR(20) | | 店铺所属品类（仅商家有效） |

> 说明：该实体承载三种角色。当role=2时，real_name作为店铺名称，shop_category记录店铺所属的8个分类之一（快餐简餐/奶茶饮品/川湘辣味/粤菜海鲜/甜品小吃/早餐早点/面食粥品/轻食沙拉）。

#### 二、实体关系分析

本系统实体之间存在以下关系：

| 关系编号 | 实体A | 关系类型 | 实体B | 关系说明 |
|----------|-------|----------|-------|----------|
| R1 | User（商家） | 一对多 | Dish | 一个商家发布多个菜品，一个菜品属于一个商家 |
| R2 | Category | 一对多 | Dish | 一个分类包含多个菜品，一个菜品属于一个分类 |
| R3 | User（用户） | 一对多 | Cart | 一个用户有多个购物车项，一个购物车项属于一个用户 |
| R4 | Dish | 一对多 | Cart | 一个菜品可被多个用户加入购物车 |
| R5 | User（用户） | 一对多 | Order | 一个用户可下多个订单，一个订单属于一个用户 |
| R6 | User（商家） | 一对多 | Order | 一个商家可收到多个订单，一个订单属于一个商家（通过shop_id） |
| R7 | Order | 一对多 | OrderItem | 一个订单包含多个订单项，一个订单项属于一个订单 |
| R8 | Dish | 一对多 | OrderItem | 一个菜品可出现在多个订单项中 |
| R9 | Order | 一对多 | OrderLog | 一个订单可有多条状态变更日志 |
| R10 | Order | 一对多 | OrderCancelLog | 一个订单可有多条取消日志 |
| R11 | User（发送者） | 一对多 | Message | 一个用户可发送多条消息 |
| R12 | User（接收者） | 一对多 | Message | 一个用户可接收多条消息 |

**关系详细说明：**

**R1 商家-菜品（一对多）：** 每个商家（role=2的User）可发布多个菜品，通过Dish.merchant_id外键关联。一个菜品只能属于一个商家。这是系统核心的商家-菜品归属关系，支撑按商家筛选菜品、商家订单隔离等业务。

**R2 分类-菜品（一对多）：** 每个菜品分类可包含多个菜品，通过Dish.category_id外键关联。一个菜品只能属于一个分类。支撑按分类浏览菜品和分类菜品数量统计。

**R3 用户-购物车（一对多）：** 每个用户可拥有多条购物车记录，通过Cart.user_id外键关联。购物车中的菜品通过dish_id间接关联到商家（shop_id），实现按店铺分组展示。

**R5 用户-订单（一对多）：** 每个用户可下多个订单，通过Order.user_id外键关联。支撑用户订单列表查询。

**R6 商家-订单（一对多）：** 每个商家可收到多个订单，通过Order.shop_id关联（shop_id存储商家的user_id）。这是美团同款多商家架构的核心关系，实现商家订单隔离——每个商家只能查看和操作自己店铺的订单。

**R7 订单-订单项（一对多）：** 一个订单包含多个订单项，通过OrderItem.order_id外键关联。订单项记录下单时的菜品快照信息（名称、单价），保证历史订单数据的完整性。

**R9 订单-状态日志（一对多）：** 一个订单的状态每次变更都会产生一条日志记录，通过OrderLog.order_id外键关联。支撑管理员查看订单状态变更历史。

**R11/R12 用户-消息（双向一对多）：** 消息实体同时关联发送者和接收者两个User实体，通过sender_id和receiver_id外键分别关联。一个用户既可以是发送者也可以是接收者，实现用户间的双向消息通信。

**E-R图关系连线说明：**

```
User ─── 1:N ─── Dish          商家发布菜品（merchant_id）
Category ─── 1:N ─── Dish      分类包含菜品（category_id）
User ─── 1:N ─── Cart          用户加购商品（user_id）
Dish ─── 1:N ─── Cart          菜品被加购（dish_id）
User ─── 1:N ─── Order         用户下单（user_id）
User ─── 1:N ─── Order         商家接单（shop_id）
Order ─── 1:N ─── OrderItem    订单包含订单项（order_id）
Dish ─── 1:N ─── OrderItem     菜品被下单（dish_id）
Order ─── 1:N ─── OrderLog     订单状态日志（order_id）
Order ─── 1:N ─── OrderCancelLog  订单取消日志（order_id）
User ─── 1:N ─── Message       用户发消息（sender_id）
User ─── 1:N ─── Message       用户收消息（receiver_id）
```

**关键设计说明：**

1. **User实体的双重角色**：同一个User实体同时承担"普通用户"和"商家"两种角色，通过role字段区分。当role=2时，该User同时作为Dish的发布者（R1关系）和Order的接收者（R6关系），实现了商家与菜品、商家与订单的关联。

2. **购物车的店铺关联**：Cart实体本身不直接存储shop_id字段，而是通过dish_id关联Dish，再由Dish的merchant_id间接关联到商家。这种设计避免了数据冗余，同时支持按店铺分组查询。

3. **订单的店铺绑定**：Order实体直接存储shop_id字段（而非通过OrderItem间接查询），这是为了提高查询效率——商家订单列表查询只需 `WHERE shop_id = ?` 即可，无需关联order_item和dish表。

4. **订单项的价格快照**：OrderItem中的dish_name和price为下单时的快照数据，即使菜品后续改名或调价，历史订单仍保留原始信息，保证财务数据的准确性。

5. **配送信息的内嵌设计**：delivery_status和delivery_position直接存储在Order实体中（而非独立的配送表），这是因为每个订单在同一时间只有一个配送状态，内嵌设计简化了查询，避免了不必要的表关联。
