# 第2章 系统设计（续）

## 2.2 系统主要功能设计

### 2.2.1 首页分类筛选商家功能

#### 功能流程说明

首页分类筛选商家功能实现了用户通过快捷分类导航快速定位目标商家的业务流程。系统首页中部展示8个美食快捷分类卡片（快餐简餐、奶茶饮品、川湘辣味、粤菜海鲜、甜品小吃、早餐早点、面食粥品、轻食沙拉），用户点击任一分类卡片后，页面跳转至商家列表页并携带对应分类参数，系统自动筛选并展示该分类下的全部商家。

具体流程如下：用户在首页点击某个分类卡片（如"川湘辣味"），前端页面拼接跳转URL为 `/merchant/list?category=川湘辣味` 并发起GET请求。MerchantListServlet控制器接收请求后，从URL参数中提取category分类值，调用UserService的findMerchantsByCategory方法执行数据库查询。DAO层构建SQL语句 `SELECT * FROM user WHERE role=2 AND status=0 AND shop_category=?` 进行条件查询，仅返回角色为商家、状态为正常且店铺分类匹配的用户记录。查询结果封装为商家列表返回给Servlet，Servlet同时查询每个商家的菜品数量，将商家列表、菜品数量映射、当前选中的分类名称传递至商家列表JSP页面渲染。页面顶部分类筛选栏中对应分类标签高亮显示，下方网格展示筛选后的商家卡片。

用户可在商家列表页点击其他分类标签进行切换筛选，也可点击"全部"标签清除筛选条件查看全部商家。整个流程体现了"首页引流→分类筛选→精准展示"的美团同款分层浏览设计。

#### 时序图描述

```
参与者：普通用户
对象：首页JSP → MerchantListServlet → UserService → UserDAO → MySQL数据库

1. 普通用户 → 首页JSP：点击"川湘辣味"分类卡片
2. 首页JSP → MerchantListServlet：GET /merchant/list?category=川湘辣味
3. MerchantListServlet → UserService：findMerchantsByCategory("川湘辣味")
4. UserService → UserDAO：查询role=2 AND status=0 AND shop_category=川湘辣味
5. UserDAO → MySQL数据库：执行SELECT SQL查询
6. MySQL数据库 → UserDAO：返回商家记录列表
7. UserDAO → UserService：返回商家列表
8. MerchantListServlet → UserService：遍历商家查询每个商家的菜品数量
9. UserService → UserDAO：执行COUNT查询
10. UserDAO → MySQL数据库：执行SELECT COUNT查询
11. MySQL数据库 → UserDAO：返回菜品数量
12. MerchantListServlet → 商家列表JSP：forward(商家列表, 菜品数量, 选中分类)
13. 商家列表JSP → 普通用户：渲染分类高亮+商家卡片网格
```

### 2.2.2 进店浏览与加入购物车功能

#### 功能流程说明

进店浏览与加入购物车功能实现了用户在商家店铺页面浏览菜品并加购的核心点餐流程。用户从商家列表页点击"进入店铺"按钮后，进入该商家的专属店铺页面，页面采用左侧分类栏+右侧菜品列表的美团同款布局。

具体流程如下：用户点击商家卡片的"进入店铺"按钮，前端发起GET请求至MerchantStoreServlet，携带shopId参数。控制器首先查询商家基本信息（通过UserService.getById），校验商家状态是否正常（role=2且status=0），若商家已被封禁则重定向至商家列表页。校验通过后，控制器查询该商家的全部上架菜品（通过DishService.findByMerchantIdFull），并按菜品分类进行分组（LinkedHashMap保持分类顺序），同时加载全部分类列表。数据传递至店铺JSP页面后，左侧渲染分类导航栏，右侧按分类依次渲染菜品列表，每款菜品展示图片、名称、描述、月销量、价格（含特价显示），以及"+"加购按钮。

用户点击菜品旁的"+"按钮时，前端发起AJAX POST请求至AddCartServlet，携带dishId和quantity=1参数。控制器从Session中获取当前登录用户ID，调用CartService.addToCart方法。该方法首先在cart表中查询该用户是否已有该菜品的购物车记录，若存在则累加数量（UPDATE），若不存在则插入新记录（INSERT）。操作完成后返回JSON响应，包含更新后的购物车总数量。前端收到成功响应后，更新菜品行的加购按钮为数量增减控件（显示当前数量），同时更新导航栏右上角的购物车角标数字。

用户可在店铺页面持续加购不同菜品，底部悬浮购物车弹窗实时显示当前店铺已选商品列表、各项小计和合计金额。点击弹窗中商品的"+""-"按钮可调整数量，点击"清空"可一键清空当前店铺购物车。

#### 时序图描述

```
参与者：普通用户
对象：商家列表JSP → MerchantStoreServlet → DishService → 菜品JSP → AddCartServlet → CartService → CartDAO → MySQL数据库

1. 普通用户 → 商家列表JSP：点击"进入店铺"按钮
2. 商家列表JSP → MerchantStoreServlet：GET /merchant/store?id=5
3. MerchantStoreServlet → OrderService/ UserService：getById(5) 校验商家状态
4. MerchantStoreServlet → DishService：findByMerchantIdFull(5)
5. DishService → DishDAO：查询merchant_id=5且status=1的全部菜品
6. DishDAO → MySQL数据库：执行SELECT SQL（含LEFT JOIN category和user）
7. MySQL数据库 → DishDAO：返回菜品列表
8. MerchantStoreServlet → 菜品JSP：forward(商家信息, 分类分组菜品)
9. 菜品JSP → 普通用户：渲染左侧分类栏+右侧菜品列表
10. 普通用户 → 菜品JSP：点击"+"按钮加入购物车
11. 菜品JSP → AddCartServlet：POST /cart/add (dishId=12, quantity=1)
12. AddCartServlet → CartService：addToCart(userId, dishId, 1)
13. CartService → CartDAO：查询购物车是否已有该菜品
14. CartDAO → MySQL数据库：SELECT * FROM cart WHERE user_id=? AND dish_id=?
15. MySQL数据库 → CartDAO：返回查询结果
16. CartService → CartDAO：INSERT或UPDATE购物车记录
17. CartDAO → MySQL数据库：执行INSERT/UPDATE
18. MySQL数据库 → CartDAO：返回操作结果
19. CartService → AddCartServlet：返回更新后的购物车总数
20. AddCartServlet → 菜品JSP：JSON {success:true, count:3}
21. 菜品JSP → 普通用户：更新加购按钮为数量控件+更新导航栏角标
```

### 2.2.3 下单结算功能（事务+跨店校验+库存扣减）

#### 功能流程说明

下单结算功能是系统最核心的业务流程，涉及跨店铺校验、数据库事务控制、库存校验与扣减、订单创建、购物车清空等多个操作的原子性保证。

具体流程如下：用户在店铺页面点击底部购物车弹窗的"去结算"按钮，前端跳转至CheckoutServlet，携带shopId参数。控制器从cart表中查询当前用户属于该店铺的购物车商品（CartService.getCartByUserIdAndShopId），计算合计金额，将购物车商品列表、总金额、店铺ID传递至结算JSP页面。用户在结算页面填写收货人姓名、手机号、收货地址，选择支付方式（微信支付或支付宝支付），点击"提交订单"。

提交后进入SubmitOrderServlet处理流程，这是整个系统最关键的事务处理环节：

**第一阶段：跨店校验。** 系统获取购物车中所有商品，遍历每件商品通过dish表查询其merchant_id，与提交的shopId进行比对。若发现任何一件商品不属于当前店铺，立即抛出异常"购物车中包含其他店铺商品，请先清空后再结算本店订单"，事务终止。

**第二阶段：事务开启。** 获取数据库连接后关闭自动提交（conn.setAutoCommit(false)），进入事务管理模式。

**第三阶段：库存校验与金额计算。** 遍历购物车商品，逐一查询dish表获取菜品价格（优先取special_price特价，否则取原价price）和当前库存stock。校验库存是否大于等于购买数量，若库存不足则抛出异常并执行事务回滚。同时累加计算订单总金额。

**第四阶段：订单创建。** 创建Order对象，生成唯一订单编号（格式：ORD + yyMMddHHmmss + 4位序列号），设置用户ID、店铺ID、收货信息、总金额、支付方式、订单状态为"待支付"（order_status=1）、创建时间。调用OrderDAO.insert方法将订单写入order表，获取自增主键orderId。

**第五阶段：订单项创建与库存扣减。** 遍历购物车商品，为每件商品创建OrderItem对象（记录菜品名称、单价、数量、小计，作为价格快照），写入order_item表。同时执行库存扣减SQL `UPDATE dish SET stock = stock - ? WHERE dish_id = ? AND stock >= ?`，若更新行数为0则表示库存不足，执行事务回滚。

**第六阶段：购物车清空与事务提交。** 所有商品处理完毕后，执行DELETE FROM cart WHERE user_id=? 清空当前用户的购物车。全部操作成功后执行conn.commit()提交事务，订单创建完成，跳转至支付页面。

**异常处理：** 任何环节抛出异常时，catch块执行conn.rollback()回滚事务，确保数据一致性。finally块中恢复自动提交并关闭连接。

#### 时序图描述

```
参与者：普通用户
对象：店铺JSP → CheckoutServlet → SubmitOrderServlet → CartDAO → DishDAO → OrderDAO → OrderItemDAO → MySQL数据库

1. 普通用户 → 店铺JSP：点击"去结算"按钮
2. 店铺JSP → CheckoutServlet：GET /order/checkout?shopId=5
3. CheckoutServlet → CartService：getCartByUserIdAndShopId(userId, 5)
4. CartService → CartDAO：查询购物车中该店铺商品
5. CartDAO → MySQL数据库：SELECT（JOIN dish表获取价格和商家ID）
6. MySQL数据库 → CartDAO：返回购物车商品列表
7. CheckoutServlet → 结算JSP：forward(购物车商品, 合计金额, shopId)
8. 结算JSP → 普通用户：渲染收货信息表单+订单摘要
9. 普通用户 → 结算JSP：填写收货信息，点击"提交订单"
10. 结算JSP → SubmitOrderServlet：POST /order/submit
11. SubmitOrderServlet → CartService：getCartByUserId(userId)获取全部购物车
12. SubmitOrderServlet → DishDAO：【跨店校验】遍历查询每件商品的merchant_id
13. DishDAO → MySQL数据库：SELECT merchant_id FROM dish WHERE dish_id=?
14. SubmitOrderServlet：校验所有商品shopId一致 → 通过
15. SubmitOrderServlet → MySQL数据库：【事务开始】SET autocommit=false
16. SubmitOrderServlet → DishDAO：【库存校验】查询价格和库存
17. DishDAO → MySQL数据库：SELECT price, special_price, stock
18. SubmitOrderServlet：库存校验通过 → 计算总金额
19. SubmitOrderServlet → OrderDAO：创建订单记录
20. OrderDAO → MySQL数据库：INSERT INTO order（获取orderId）
21. SubmitOrderServlet → OrderItemDAO：循环创建订单项（价格快照）
22. OrderItemDAO → MySQL数据库：INSERT INTO order_item
23. SubmitOrderServlet → DishDAO：扣减库存
24. DishDAO → MySQL数据库：UPDATE stock = stock - ?（校验stock>=?）
25. SubmitOrderServlet → CartDAO：清空购物车
26. CartDAO → MySQL数据库：DELETE FROM cart WHERE user_id=?
27. SubmitOrderServlet → MySQL数据库：【事务提交】COMMIT
28. SubmitOrderServlet → 支付JSP：redirect(/order/pay?id=orderId)
29. 支付JSP → 普通用户：显示支付页面
```

### 2.2.4 订单状态与配送管理功能

#### 功能流程说明

订单状态管理功能实现了美团同款的多角色订单状态流转机制，涵盖商家接单处理和配送信息管理两大子流程。

**商家接单处理流程：** 商家在订单列表页查看本店订单（通过AdminOrderServlet加载，自动按shopId过滤）。订单列表支持按用户名搜索和排序（默认排序：未完成优先+时间倒序；按时间排序：纯时间倒序）。对于"待接单"（order_status=2）状态的订单，商家可执行两种操作：

（1）接单操作：商家点击"接单"按钮，前端发起AJAX POST请求至AdminOrderServlet，携带action=accept参数。控制器调用OrderService.merchantAccept方法，首先校验订单当前状态是否为"待接单"，校验通过后调用OrderDAO.updateStatusWithAcceptTime将订单状态更新为"已接单"（order_status=4）并记录接单时间（accept_time），同时写入order_log状态变更日志。接单时间用于后续用户取消订单的时间判断（1分钟内可直接取消）。

（2）拒单操作：商家点击"拒单"按钮，前端弹出原因输入框，用户填写拒单原因后提交。控制器调用OrderService.merchantReject方法，将订单状态更新为"已取消"（order_status=9），同时将拒单原因写入cancel_reason字段和order_log日志。

**已接单/制作中状态操作：** 商家可将"已接单"状态更新为"制作中"（order_status=5），或将"制作中"状态更新为"配送中"（order_status=6）。在已接单或制作中状态下，商家也可主动取消订单（需填写原因）。系统校验状态合法性后执行更新并记录日志。

**管理员强制操作：** 管理员拥有最高权限，可将任意订单状态强制变更为任意目标状态。执行时系统强制要求填写操作原因（非空校验），调用OrderService.adminForceStatus方法完成状态更新，并将完整的变更记录（原状态→新状态、操作人、原因、时间）写入order_log表。

**配送信息管理流程：** 订单进入"配送中"状态后，商家或管理员可在订单详情页更新配送信息。前端展示配送状态选择框（待配送/配送中/已送达）和位置描述输入框。提交后调用DeliveryUpdateServlet，系统首先校验配送状态是否已为"已送达"（已送达则锁定不可修改），再校验商家操作权限（仅能修改自己店铺的订单），校验通过后调用OrderDAO.updateDelivery更新delivery_status和delivery_position字段。

用户端通过DeliveryQueryServlet查询配送信息，系统返回配送状态文本和位置描述，前端以弹窗形式展示。

#### 时序图描述

```
参与者：商家
对象：订单列表JSP → AdminOrderServlet → OrderService → OrderDAO → OrderLogDAO → MySQL数据库

1. 商家 → 订单列表JSP：访问订单列表页
2. 订单列表JSP → AdminOrderServlet：GET /admin/order
3. AdminOrderServlet → OrderService：getOrdersByMerchantId(merchantId, keyword, sort)
4. OrderService → OrderDAO：查询该商家的全部订单（JOIN order_item筛选）
5. OrderDAO → MySQL数据库：执行SELECT SQL
6. MySQL数据库 → OrderDAO：返回订单列表
7. AdminOrderServlet → 订单列表JSP：forward(订单列表, 商家列表)
8. 订单列表JSP → 商家：渲染订单表格（含接单/拒单按钮）

9. 商家 → 订单列表JSP：点击"接单"按钮
10. 订单列表JSP → AdminOrderServlet：POST /admin/order/status (action=accept, orderId=10)
11. AdminOrderServlet → OrderService：merchantAccept(10, merchantId)
12. OrderService → OrderDAO：findById(10) 校验当前状态为"待接单"
13. OrderDAO → MySQL数据库：SELECT order_status FROM order WHERE order_id=10
14. OrderService → OrderDAO：updateStatusWithAcceptTime(10, 4)
15. OrderDAO → MySQL数据库：UPDATE order SET order_status=4, accept_time=NOW()
16. OrderService → OrderLogDAO：insert(日志记录)
17. OrderLogDAO → MySQL数据库：INSERT INTO order_log
18. AdminOrderServlet → 订单列表JSP：JSON {success:true}
19. 订单列表JSP → 商家：刷新页面，订单状态更新为"已接单"
```

### 2.2.5 AI客服调用功能

#### 功能流程说明

AI智能客服功能通过集成通义千问大模型API，为用户提供基于真实订单数据的智能问答服务。该功能的核心设计在于"上下文感知"——系统自动将用户的订单和配送信息作为上下文传递给AI模型，使AI能够基于真实数据回答问题，而非泛泛而谈。

具体流程如下：用户点击页面右下角紫色悬浮按钮展开聊天窗口，在输入框中输入问题（如"我的外卖到哪了"）并发送。前端通过XMLHttpRequest发起POST请求至AiOrderChatServlet，携带question参数。

AiOrderChatServlet控制器首先校验用户登录状态（从Session获取用户信息），未登录则返回错误提示。校验通过后，系统执行上下文构建：

（1）查询当前用户的全部订单：调用OrderService.getOrdersByUserId获取用户的所有历史订单列表。

（2）遍历订单构建上下文文本：对每个订单提取订单号（orderNo）、订单状态文本（通过getOrderStatusText方法转换状态码为中文：1→待支付、2→待接单、4→已接单、5→制作中、6→配送中、8→已完成、9→已取消）、订单金额、收货地址、配送状态文本（0→待配送、1→配送中、2→已送达）、当前配送位置。

（3）拼接完整消息：将订单上下文和用户问题组合为结构化文本，格式为：
```
【用户当前订单信息】
订单号：ORD2606151230390004，状态：配送中，金额：36.00，地址：Beijing Chaoyang District，配送：待配送，位置：暂无

【用户问题】：我的外卖到哪了
```

（4）调用AI接口：AiChatUtil.chat方法通过HttpURLConnection发起POST请求至通义千问DashScope兼容模式API（https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions），请求体包含system角色的系统提示词（约束AI行为）和user角色的拼接消息。API Key和模型名称（qwen3.6-flash）存储在AiConfig配置类中，不暴露至前端。

（5）解析AI响应：从API返回的JSON中提取content字段作为AI回答，进行JSON转义处理后返回前端。

（6）前端渲染：聊天窗口以气泡形式展示AI回复，用户消息居右（紫色渐变背景）、AI消息居左（灰色背景），聊天区域自动滚动至最新消息。

**系统提示词约束：** AI被设定为美食点餐系统专属客服，仅解答订单查询、配送进度、取消规则、退款、菜品咨询等相关问题，必须依据传入的真实订单数据回复，不得编造信息，无关闲聊统一回复固定话术，语言口语化简洁，回复不超过100字。

#### 时序图描述

```
参与者：普通用户
对象：聊天窗口JSP → AiOrderChatServlet → OrderService → OrderDAO → AiChatUtil → 通义千问API → MySQL数据库

1. 普通用户 → 聊天窗口JSP：点击悬浮按钮展开聊天窗口
2. 普通用户 → 聊天窗口JSP：输入"我的外卖到哪了"并点击发送
3. 聊天窗口JSP：显示"正在思考中..."加载提示
4. 聊天窗口JSP → AiOrderChatServlet：POST /aiOrderChat (question=我的外卖到哪了)
5. AiOrderChatServlet：校验Session中用户登录状态 → 通过
```
